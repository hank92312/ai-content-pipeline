# 專案名稱：AI 自動化短影音內容流水線 (AI Content Pipeline)

## 📌 專案與頻道核心設定
* **專案目標**：建立一套半自動化的資料處理系統，從網路抓取最新財經與科技硬體新聞，透過 AI 生成口語化的短影音腳本，最終結合虛擬角色、AI 繪圖與自動剪輯，產出可一鍵發布的短影音。
* **頻道名稱**：小花的AI情報站
* **角色設定**：蔡小花 (⚠️ **核心註記：角色性別為男性**。以真人形象轉化之平面二次元/科技感風格，講述硬核財經與 3C 情報)
* **影片規格**：直式短影音 (Shorts/Reels 格式)，總長度約 30~50 秒。

## 🛠 開發環境與技術棧
* **主要語言**：Python 3.11 (虛擬環境 .venv)
* **資料庫**：SQLite (`auto_channel.db`)
* **AI 文字大腦**：Google Gemini API (`gemini-2.5-flash`)
* **AI 聲音大腦**：使用 `edge-tts` (語速 +25%) 產出基礎音檔，並導入 RVC (VITS 架構) 進行音色轉換。已實作 AMD/Intel GPU (DirectML) 加速，並具備 CPU 自動降級補償。

## 📂 系統架構與模組說明

### 模組一：資料蒐集與儲存 (Data Collection)
負責從外界獲取原始資訊，並統一結構化儲存至本地資料庫中。
* **`auto_channel.db` (資料庫)**：系統核心大腦，包含 `DailyNews` 資料表。具備狀態管理欄位：`is_selected` (總編輯已挑選), `is_processed` (已產製成腳本)。
* **`step1_scraper.py` (自動爬蟲)**：利用 `feedparser` 自動抓取設定好的 RSS 來源（如：巴哈姆特、科技新報、Yahoo 財經）。內建 `UNIQUE link` 防重複機制，可設定每日排程。
* **`step1_manual_add.py` (手動靈感收集)**：利用 `BeautifulSoup` 輔助，貼上網址即可半自動抓取網頁標題，並手動分類寫入資料庫。

### 模組一點五：人機協作過濾 (Human-in-the-Loop)
* **`step1_5_selector.py` (總編輯選題台)**：撈取資料庫中尚未處理的新聞清單，讓使用者輸入欲製作成影片的新聞 ID，並將其在資料庫中標記為 `is_selected = 1`。

### 模組二：AI 劇本生成 (AI Script Generation) - [已產製，具備 30 秒限字與雙配圖格式]
負責將生硬的新聞標題轉化為生動的短影音口白。
* **`step2_script_generator.py`**：
  * **功能**：自動讀取 `is_selected = 1` 的新聞，改為**每一則新聞獨立生成專屬腳本**。
  * **AI 處理**：呼叫 `gemini-2.5-flash`，要求產出 JSON 格式，字數嚴格限制在 100~120 字 (對應約 30 秒)，並強制提供 2 組 `visual_prompts`。
  * **動態頭尾邏輯 (重點)**：需透過 Prompt 要求 AI 根據新聞內容，動態生成「三秒內抓住眼球」的開場白，以及結尾「自然引導互動（如提問、引導留言與訂閱）」的收尾句。
  * **產出**：純文字檔與 JSON 中繼檔存入 `output_scripts/` 資料夾，並將資料庫狀態更新為 `is_processed = 1`。

### 模組三：語音渲染中心 (Voice Rendering) - [已產製，支援 RVC 音色轉換]
負責將文字腳本轉化為高品質的語音資源，並生成時間軸資料。
* **`step3_voice_renderer.py`**：
  * **文字轉語音 (TTS)**：讀取 `output_scripts/` 中的腳本，使用 `edge-tts` 將文字轉換為基礎 `.wav` 音檔。語速調整為 **+25% (約 1.2 倍速)**，並同步產出包含字級時間軸的 `_subs.json` 中繼檔。
  * **聲音轉換 (Voice Conversion)**：已導入 RVC 模型「蔡小花」音色。推理過程優先嘗試使用 **AMD GPU (DirectML)** 加速，若硬體不支援則自動回退至 **CPU 模式**。
  * **最終產出**：產出 `*_final.wav` (最終音色) 與 `*_subs.json` (字幕時間軸) 存於 `output_voices/`。

### 模組四：虛擬角色創作 (Virtual Avatar) - [初版使用靜態佔位圖]
* **視覺呈現**：目前專案根目錄已具備 `sample.png` 靜態去背人物圖。
* **動態驅動**：不追求 3D 建模。將平面角色圖透過 AI 驅動模組（如 SadTalker、Live2D 腳本或其他 Wav2Lip 技術），達成播報時右下角頭像能配合語音產生「嘴型同步、臉部微表情、偶爾手部動作」之效果。

### 模組五：新聞簡報圖 (Visual Assets) - [已產製，支援雙配圖生成]
* **素材獲取**：支援雙配圖。第一張透過 Gemini Vision 解析新聞原始圖片重製；第二張直接依據腳本後半段的 `visual_prompts` 純提示詞生成。
* **版權處理 (風格化)**：為避免版權爭議，將透過 Gemini Vision 解析原始圖片，再交由 Imagen 強制套用二次元/科技感風格。**已嚴格禁止生成任何中英文字母或亂碼**，確保畫面純淨。
* **版面規格**：符合直式影片比例 (9:16)。自動分配圖片長度。

### 模組六：影片自動生成與字幕壓製 (Video Assembly & Subtitles) - [已產製，具備硬體加速]
* **雙模式素材合成 (雙軌並行)**：
  1. **原模式 (預設)**：利用 `output_images` 自動生成之配圖與配音，動態對齊合成。
  2. **素材模式**：從 `assets` 資料夾抓取「同新聞ID」之 `.mp4` (循環背景)、`.jpg` (結尾轉場)、`.mp3` (低音量背景樂) 進行專業模板產製。
* **自動剪輯與字幕**：`step6_video_assembly.py` 利用 `MoviePy` 將素材動態對齊合成。同時支援讀取時間軸與動態畫筆，直接在畫面上繪製高彩度字幕。
* **重點標色**：判讀腳本提供的 `keywords` 陣列，動態對字幕標記重要字眼為「電競黃色」或「警示紅色」。
* **硬體加速**：輸出壓製時支援 **AMD AMF (h264_amf)** 與 **Intel QuickSync (h264_qsv)** 硬體編碼器，失敗時自動無縫降級為 **CPU (libx264)**。
* **產出**：最終只需渲染一次，即輸出 9:16、帶有字幕之 `*_subtitled.mp4` 短影音至 `output_videos/`。

### 模組八：多平台發布 (Multi-Platform Distribution) - [✅ 已完成]
* **核心技術**：使用 **Playwright** 瀏覽器自動化模擬人工發布，解決 API 申請門檻。
* **AI 行銷文案**：連動 Gemini API 根據腳本內容自動生成吸睛標題與 Hashtags。
* **發布平台進度**：
  - **YouTube Shorts**：✅ 已完成全自動發布。
  - **Facebook Reels**：✅ 已完成全自動發布。
  - **Instagram (IG) Reels**：✅ 已完成全自動發布 (含 9:16 比例切換)。
  - **TikTok**：✅ 已完成全自動發布 (支援版權檢查跳過)。
* **自動過濾**：系統會自動對照資料庫，僅顯示尚未「歸檔」的影片，避免重複發布。

### 模組九：完工歸檔與環境整理 (Archiving & Cleanup) - [✅ 已完成]
負責將發布成功的作品進行系統化存檔，並保持開發環境整潔。
* **`step9_manual_archive.py` (手動歸檔)**：
  - **歸檔展示**：展示所有已產製但未歸檔的影片清單。
  - **多重資產搬移**：確認發布成功後，一鍵將「影片 (字幕版)、劇本 (JSON/TXT)、語音 (含 final/tts/subs)、配圖」整組搬移至 `completed_archives/`。
  - **自動空間優化**：歸檔同時會自動刪除 `output_videos/` 下原始「無字幕版本」的影片檔，以節省存儲空間。
  - **環境相容性**：內建 UTF-8 強制編碼轉換，確保在 Windows 終端機 (cp950) 環境下也能正常顯示圖示與中文。
  - **狀態變更**：在資料庫標記為 `is_published = 1`，正式結案。
* **架構優化 (Optimization)**：
  - **配置中央化**：透過 `config.py` 統一管理全域路徑與 AI 參數。
  - **安全性強化**：引入 `.env` 檔案隔離敏感 API Keys，不直接寫死在程式中。
  - **自動清理**：偵錯過程產生的臨時圖檔與 HTML 統一導向 `debug_logs/`，保持根目錄清爽。

## 🛑 開發與測試守則
1. **API 成本管控**：每次進行流程測試時（如生圖、生劇本），請先將資料表的 `is_selected` 控制在 **1 筆**，確保一次只生成**一篇短影音**，避免消耗過多 Gemini API 額度並節省除錯時間。
2. **語音斷點優化**：過渡期採用的 `edge-tts` 在斷句轉換上雖有停頓與不連貫感，將不刻意修復，待後續 RVC 模型導入時再予以根本性修正。
3. **硬體相容性優先**：所有高效能運算模組 (RVC/FFmpeg) 必須維持「優先嘗試 GPU 加速 (支援 AMD AMF, Intel QSV, DirectML) -> 失敗回退 CPU」的邏輯，確保專案可在筆電與桌電間無縫切換。
4. **環境變數管理**：所有敏感的金鑰 (如 `GEMINI_API_KEY`) 必須儲存在根目錄的 `.env` 中，嚴禁硬編碼。
5. **專案備份與同步**：詳細步驟請參閱根目錄 `SOP.md`。程式碼透過 GitHub 私人中心同步，模型與 Session 則採手動搬運。
